# 開発規律 (Contribution Guide)

## 1. ブランチ戦略 (GitHub Flow)
- **`main`**: 常にデプロイ可能な、動作確認済みのコードのみを保持。
- **`feature/xxxx`**: 新機能開発、バグ修正、ドキュメント更新用の作業ブランチ。
  - 原則として、Pull Request (PR) を通じて `main` へマージする。

## 2. コミットメッセージ規約 (Conventional Commits)
以下のプレフィックスを必須とする。
- `feat:` 新機能の実装
- `fix:` バグ修正
- `docs:` ドキュメントのみの更新
- `refactor:` 機能を変えないコードの整理
- `test:` テストコードの追加・修正

## 3. プルリクエスト (PR) ルール
- PR作成時に、関連するIssueを紐付ける（`Closes #1` 等）。
- マージ前に必ず「セルフレビュー」を行い、コードの意図をコメントに残す。

## 4. ドキュメント優先 (Docs as Code)
- 実装変更時には、必要に応じて `standards` リポジトリの規約やADR（設計決定記録）を更新すること。
