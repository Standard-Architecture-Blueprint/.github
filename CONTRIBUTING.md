# 開発規律 (Contribution Guide)

本プロジェクトでは、開発プロセスの透明性と成果物の品質を一貫させるため、以下の開発標準を定める。

## 1. ワークフロー (GitHub Flow)
全ての変更は、以下のプロセスを経てメインコードに統合される。

1. **Issueの作成**: 全ての作業はIssueから開始し、目的（WHY）と完了条件を明確にする。
2. **ブランチの作成**: `main` ブランチから作業用ブランチを切り出す。
   - 命名規則: `feature/issue-[番号]-[作業概要]` （例: `feature/issue-5-setup-infra`）
3. **コミット**: 作業単位でこまめにコミットする（規約は後述）。
4. **Pull Request (PR) の作成**: 作業完了後、`main` に向けたPRを作成する。
5. **レビューとマージ**: セルフレビューまたは相互レビューを経て、`main` へマージする。

## 2. コミットメッセージ規約 (Conventional Commits)
変更履歴のトレーサビリティを確保するため、[Conventional Commits](https://www.conventionalcommits.org/) に準拠した形式を採用する。

`型: 内容の要約（日本語可）`

- **feat**: 新機能の実装
- **fix**: バグの修正
- **docs**: ドキュメントの作成・更新
- **refactor**: リファクタリング（機能変更を伴わないコード整理）
- **test**: テストコードの追加・修正
- **chore**: ビルドプロセスやライブラリの更新など

## 3. プルリクエスト (PR) のルール
PR作成時には以下の項目を確認すること。
- **目的の明示**: `Closes #1` のように、対応するIssue番号を説明文に含める。
- **設計基準の遵守**: 別リポジトリ `standards` で定義された設計規約に準拠していること。
- **デプロイ可能性**: `main` ブランチは常にビルド可能で、テストが通過する状態を維持すること。

## 4. ドキュメントの維持
「コードとドキュメントの一致」を重視する。機能変更時には、必ず `README.md` や関連する設計書を更新し、情報の鮮度を保つこと。
