# 開発規律 (Contribution Guide)

本プロジェクトでは、開発プロセスの透明性と成果物の品質を一貫させるため、AP試験の「共通フレーム（SLCP）」および「ソフトウェア品質特性」に基づいた開発標準を定める。

## 1. ワークフロー (GitHub Flow & Change Management)

全ての変更は、構成管理の原則に基づき、以下のプロセスを経てメインコードに統合される。

1. **Issueの作成**: 全ての作業はIssueから開始し、目的（WHY）と完了条件（DoD）を明確にする。
2. **ブランチの作成**: `main` ブランチから作業用ブランチを切り出す。
   - 命名規則: `feature/issue-[番号]-[作業概要]` （例: `feature/issue-5-setup-infra`）
3. **実装とコミット**: 作業単位で論理的な区切りごとにコミットする。
4. **Pull Request (PR) の作成**: 作業完了後、`main` に向けたPRを作成する。
5. **レビューとマージ**: 
   - **机上レビュー**: ソースコードの論理的整合性を確認する。
   - **テスト**: 修正箇所だけでなく、既存機能への影響（退行/デグレ）がないかを確認する。
   - 承認を経て、`main` へマージ（統合）する。

## 2. コミットメッセージ規約 (Conventional Commits)

変更履歴のトレーサビリティ（追跡可能性）を確保するため、以下の形式を採用する。

`型: 内容の要約（日本語可）`

- **feat**: 新機能の実装（機能要件の追加）
- **fix**: バグの修正（不具合の解消）
- **docs**: ドキュメントのみの変更
- **style**: コードの動作に影響しない形式の変更（スペース、書式等）
- **refactor**: リファクタリング（機能変更を伴わない内部構造の改善）
- **test**: テストコードの追加・修正
- **chore**: ビルドプロセスやライブラリの更新、環境設定など

## 3. プルリクエスト (PR) のルール

PR作成時には、AP試験の「ソフトウェア検証」の観点から以下の項目を必須とする。

- **Issue連携**: `Closes #1` のように、対応するIssue番号を必ず含める。
- **変更内容の要約**: 何を、なぜ、どのように変更したかを記述する。
- **設計基準の遵守**: リポジトリ `standards` で定義された設計規約に準拠しているか。
- **非機能要件の考慮**: パフォーマンスやセキュリティ（機密性・完全性）を損なっていないか。
- **ビルド・テストの成功**: `main` は常にリリース可能な状態（Production Ready）を維持すること。

## 4. ドキュメントと構成管理

「コードとドキュメントの一致」は、保守性の向上に直結する。

- **同時更新の原則**: 機能変更時には、必ず `README.md` や関連する設計書を更新し、情報の鮮度を保つこと。
- **ADR (Architecture Decision Records)**: 重要な技術的決定（トレードオフを伴う選択）を行った場合は、その理由をドキュメントに残すこと。
